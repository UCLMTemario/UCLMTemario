De computación “física” a entornos cloud

Virtualización

Imaginemos un ordenador. Cuando lo encendemos, el Sistema Operativo se carga. El Sistema Operativo gestiona y es capaz de actuar sobre el hardware del equipo. Posteriormente, nosotros podremos ejecutar aplicaciones sobre el Sistema Operativo.

Con la virtualización, una aplicación conocida como hipervisor se carga como cualquier otra aplicación.

Este hipervisor es capaz de emular cualquier número de ordenadores virtuales, cada uno son su propio Sistema Operativo.

Además, gestiona el acceso de la máquina con el hardware del ordenador.

  • Tipo 1: No se emplea ningún tipo de sistema operativo. Se podría decir que el hipervisor se ejecuta directamente sobre el hardware del equipo y crea distintas máquinas virtuales.

  • Tipo 2: Se ejecuta el hipervisor sobre el sistema operativo huésped.

También existe la paravirtualización, donde el Sistema Operativo abstrae recursos hardware para múltiples entornos virtuales sin tener que virtualizar el hardware para cada huésped.

  • Aumento de eficiencia y rendimiento.

La virtualización atrajo la atención en las compañías cuando se dieron cuenta que los servidores que adquirían se encontraban infrautilizados.

Las grandes empresas necesitan dar soporte a cientos de configuraciones de infraestructuras diferentes.

  • Número menor de servidores.

  • Mejor gestión de configuración.

  • Adición y eliminación de recursos.

  • Copiar y trasladar fácilmente.

Aunque la virtualización es concepto clave para hacer cloud computing, se debate en si es suficiente para considerarse cloud.

  • Infraestructura como servicio.

  • Plataforma como servicio.

  • Software como servicio.

Infraestructura como servicio

La capacidad que se proporciona al consumidor es la de suministrar procesamiento, almacenamiento, redes y otros recursos informáticos fundamentales en los que el consumidor puede desplegar y ejecutar software arbitrario, que puede incluir sistemas operativos y aplicaciones.

Plataforma como servicio

La capacidad que se proporciona al consumidor es la de desplegar en la infraestructura de la nube aplicaciones adquiridas por el consumidor o creadas utilizando lenguajes de programación, bibliotecas, servicios y herramientas soportadas por el proveedor.

Software como servicio

La capacidad que se ofrece al consumidor es la de utilizar las aplicaciones del proveedor que se ejecutan en una infraestructura en la nube.

El consumidor no gestiona ni controla la infraestructura subyacente.

Infraestructura como código

Aumento de la necesidad de replicar servidores idénticos y de auto-escalado.

No es posible tener un trabajador continuamente configurando y habilitando tales infraestructuras.

Gracias a la virtualización, es muy fácil crear y destruir servidores.

Rutinas coherentes y repetibles para el aprovisionamento y la modificación de los sistemas y su configuración.

Los cambios se introducen en las definiciones y luego se aplican a los sistemas mediante procesos desatendidos que incluyen una validación exhautiva.

Emplear técnicas ágiles como Test Driven Development, Continuous Integration y Continuous Delivery no permite modificar la infraestructura de manera rápida y eficaz.

Dado que los cambios son el mayor riesgo para un sistema de producción, el cambio es inevitable y para mejorar hay que cambiar; tiene sentido optimizar su capacidad para realizar cambios de forma rápida y fiable.

Aprovecharemos el dinamismo de las infraestructuras modernas para realizar cambios frecuentes y fiables.

  • Definir todo como código.

  • Entregar y testear continuamente todo el trabajo en curso.

  • Desarrollar piezas pequeñas y sencillas que se puedan cambiar indpendientemente.

Definir todo como código

Gestionar todos los aspectos del desarrollo, la entrega y la gestión del software mediante la definición y codificación de la infraestructura, el esquema y las tuberias utilizados para crear, mantener, iterar o ampliar el desarrollo de aplicaciones.

Aplicar el enfoque de desarrollo de aplicaciones a otros componentes de Tecnologías de Información para garantizar que las mejoras prácticas se definan y se sigan con un mínimo de esfuerzo.

Definir todo el material “como código” es una práctica fundamental para realizar cambios de forma rápida y fiable.

  • Reutilización: Si defines una cosa como código, puedes crear muchas instancias de ella.

  • Consistencia: Las cosas construidas a partir de código se construyen siempre de la misma manera.

  • Transparencia: Todo el mundo puede ver cómo se construye el objeto mirando el código.

Entregar y testear continuamente

Utilizar la automatización para desplegar y probar cada componente de su sistema, e integran todo el trabajo que todos tienen en marcha.

La idea es crear calidad en lugar de testearla.

  • Integrar y probar todo el trabajo en curso.

Desarrollar piezas pequeñas y sencillas

Cuánto más grande es un sistema, más fácil es cambiarlo y más fácil es romperlo.

Dividir en componentes pequeños es fácil de entender y tiene interfaces claramente definidas para cambiar fácilmente cada componente y desplegar y probar cada componente de forma aislada.

Gestión de la configuración

Práctica de importancia crítica en los sistemas digitales.

El seguimiento y el control de sus productos de trabajo a medida que evolucionan a través de un cambio tras otro es importante desde el primer día.

  • Hacer una copia de seguridad o archivar el estado de un sistema.

  • Comparar dos versiones del estado del sistema e identificar las diferencias.

  • Restaurar el sistema a un estado operativo.

Control de versiones

Se considera como uno de los cuatro pilares de las metodologías ágiles; junto con el equipo, el desarrollo iterativo y el desarrollo incremental.

Es crítico para cualquier tipo de sistema con contenido complejo y cambiante.

Gestión de paquetes

Se emplea para gestionar el estado de la infraestructura.

Distinción entre source control y gestión de paquetes: la gestión de archivos binarios, a diferencia de los archivos simbólicos comprensibles para el ser humano.

Gestión de despliegue

Los recursos en control de versiones aún no están entregando valor; su combinación con recursos informáticos sí lo hará.

La gestión de despliegue administra la combinación de los artefactos del control de versiones con los recursos necesarios para la entrega de valor.
