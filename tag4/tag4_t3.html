Sistemas Gestores de Bases de Datos

Componentes

  • Motor de la base de datos: acepta peticiones lógicas de los otros subsistemas del SGDB, las convierte en su equivalente físico y accede a la base de datos en el dispositivo de almacenamiento.

  • Subsistema de definición de datos: define la estructura del fichero que soporta la base de datos.

  • Subsistema de manipulación: ayuda al usuario a añadir, cambiar y borrar información de la base de datos y la consulta para extraer información.

  • Subsistema de generación de aplicaciones: contiene utilidades para ayudar a los usuarios en el desarrollo de aplicaciones.

  • Subsistema de administración: ayuda a gestionar la base de datos ofreciendo funcionalidades de almacenamiento y recuperación, gestión de la seguridad, optimización de preguntas, control de concurrencia y gestión de cambios.

Evolución

    • Basados en datos en red.

    • Sistemas open source.

  3. sistemas post-relacionales.

    • Sistemas orientados a objetos.

    • Sistemas objeto-relacionales.

BD Distribuidas y Teorema CAP

Bases de Datos Distribuidas

Colección de datos lógicamente interrelacionada de datos compartidos distribuidos en diferentes nodos de una red de ordenadores.

  • Globalización.

  • Capacidad computacional.

  • Compartir datos.

Esquemas

  • Esquema conceptual global: descripción lógica de toda la base de datos, como si no estuviera distribuida.

  • Esquemas de fragmentación y asignación: cómo hay que partir los datos, cómo hay que ubicarlos teniendo en cuenta los mecanismos de replica.

  • Esquemas locales: descripción lógica de la base de datos local.

Aspectos relevantes

  • Diseño centralizado.

  • Proceso de distribuición de datos.

  • Cómo dividir la base de datos en componentes para localizarlos en diferentes sitios.

  • Qué cantidad de datos debe ser replicados y cómo deben los fragmentos replicados ser localizados.

  • Cada ordenador debe tener su propia base de datos y debe estar capacitada para acceder a los datos de otros ordenadores.

  • Los datos suelen almacenarse en un lugar próximo a aquel donde más frecuentemente son utilizados.

Control centralizado Control jerárquico
Independencia de datos: organización de los datos transparente para el progamador Transparencia en la distribución

Seguridad Problemas de seguridad intrínsecos

Teorema CAP

  1. Consistencia: se refiere a sí los datos tienen igual versión.

  2. Disponibilidad: se refiere al hecho de que cada petición recibida debe obtener una respuesta.

  3. Tolerancia a fallos: la base de datos no deja de funcionar a pesar de que caiga un servidor o se corten conexiones de red entre servidores.

Sólo dos de cada tres pueden ser consideradas a la vez.

Bases de Datos NoSQL

¿Por qué surge la necesidad de Big Data?

Velocidad

En Internet la velocidad de respuesta de los sistemas tiene repersución directa con la satisfacción del cliente y los ingresos.

Escalabilidad

  • Las compañías necesitan mantener la respuesta rápida aunque se incremente el número de usuarios simultaneos y/o el volumen de datos.

  • Se ejecutan en clústeres de máquinas baratas.

Arquitectura

  • Escala sin disminuir el rendimiento.

  • Añade nodos sobre la marcha sin interrupciones del servicio.

  • No genera cuellos de botella.

¿Por qué surge la necesidad de Big Data?

Independencia de localización

La globalización del mercado en WWW obliga a dar servicio rápido y en todas partes del mundo.

Disponibilidad

  • Es lo más grave, por las repercusiones en los ingresos.

  • Hay que pasar alta disponibilidad a continua disponibilidad.

Carácterísticas

  • Diseño sin servidor maestro.

  • Multi-data center.

  • Disponibilidad cloud.

  • Copias de datos y funcionalidad en múltiples localizaciones.

Tipos de datos

  • Necesidades nuevas: datos estructurados, semiestructurados y no estructurados.

  • Perspectiva eficiente de almacenamiento.

Ventajas y desventajas

Ventajas Desventajas
Optimizado para recuperación y agregación Pérdida de flexibilidad en tiempo de ejecución

Características

  • No relacional.

  • Distribuida.

  • Código abierto.

  • Libre de esquemas.

De ACID al teorema CAP

Atomic

Las transacciones se ejecutan completamente o no se hace.

  • Si fallan es como si ni siquiera se hubieran intentado ejecutar.

Consistency

Garantiza que cualquier transacción llevará la base de datos de un estado válido a otro estado válido.

  • Cualquier dato que se escriba en la base tiene que ser válido de acuerdo con todas las reglas de integridad definidas en el modelo.

Isolation

Cuando varias transacciones se ejecutan en paralelo, cada una de ellas ve el sistema de la misma manera a como lo vería si se ejecutaran de forma aislada o secuencial.

Durability

El resultado de las transacciones es un cambio en el estado del sistema persistente.

  • Si apagamos la máquina y la arrancamos de nuevo el cambio producido por la transacción aún está, persiste.

Teorema CAP en NoSQL

Tolerancia a fallos (P) y consistencia (C)

Poner muchos nodos y replicar en todos.

  • Degrada la disponibilidad (A).

Aumentar la disponibilidad (D)

No esperar a replicar todos los nodos.

  • Degrada la consistencia (C).

Abandonar la tolerancia a fallos (P)

Un solo nodo.

  • Consigue consistencia (C) y disponibilidad (A).

Limitaciones

Limitaciones técnicas

  • Cómo modelar correctamente los datos para maximizar las capacidades.

  • Nivel bajo de seguridad.

  • No soporta transacciones.

  • Falta de madurez en Business Intelligence.

  • Problemas de compatibilidad ya resueltos en los modelos relacionales.

Limitaciones no técnicas

  • Falta de expertos.

  • Resistencia al cambio.

  • Disponibilidad del vendedor.

  • El código abierto puede implicar problema de soporte para las empresas.

Tipos

Clave-valor

Almacena datos en pares clave-valor, donde cada clave es única y se asocia con un valor.

  • El dato se accede a partir de una clave única.

  • Los valores de los datos son independientes y no están relacionados entre ellos.

  • Ofrece acceso rápido y eficiente a los datos a través de la clave.

  • La estructura suele venir establecida por la aplicación.

Documental

Almacena datos semi-estructurados.

  • Los datos se llaman documentos y pueden estar XML, JSON, binary JSON o el que acepte la misma base de datos.

  • Cada documento puede contener datos animados y es auto-contenible, lo que significa que puede tener su propio esquema.

  • Los documentos pueden representar objetos complejos y su relación con otros documentos.

Basado en grafos

Almacena datos semi-estructurados.

  • Basado en teoría de grafos: los nodos son la información y las aristas son las relaciones.

  • Consultas: recorridos sobre grafos.

  • Facilita la representación de relaciones complejas.

  • Óptimo para consultas que involucran patrones de relaciones.

Orientado a columnas

Almacena datos en columnas en lugar de filas.

  • Cada fila puede tener una configuración de columna distinta.

  • Permite una lectura eficiente de conjuntos de datos específicos.

  • Óptimo para consultas analíticas y agragaciones en grandes cantidades de datos.

Filas vs columnas

Filas Columnas
Ventaja Eficiencia insertar y eliminar Eficiencia de acceso a datos útiles
Desventaja Lectura de datos inútiles Ineficiencia insertar y eliminar

Multi-modelo

Evolución para admitir varios de estos modelos de datos simultáneamente.

  • Útil en situaciones donde los datos tienen estructuras variadas y diferentes necesidades de consulta.

  • Flexibilidad y simplificación en el desarrollo al permitir utilizar el modelo de datos más adecuado para cada caso de uso sin tener que recurrir a múltiples SGDB.

Tipos de almacenamiento en bases de datos NoSQL

Tecnologías de Rastreo

Aquellas para el procesamiento, indexación y recuperación para conseguir acceso eficiente a contenidos multi-formato.

Objetivo de la indexación

  • Acelerar el acceso a los datos.

  • Funciones básicas: Put, Update, Get, Delete.

  • Búqueda o consultas.

Requisitos de indexación en Big Data

  • Rápidez de búsqueda: búsqueda sobre bitones-trillones de valores de datos en segundos.

  • Consultas multi-variable: ser eficiente al combinar resultados de variables individuales.

  • Tamaño del índice: debe ser una fracción del dato original.

  • Paralelización: debe ser fácilmente particionable en partes para hacer procesamiento paralelo.

  • Velocidad en la generación de índices: para el procesamiento in situ, el índice se tiene que construir a la misma vez que se procesa el dato.

Estructura de Índices de datos

Los datos se almacenan en unidades de acceso fijo.

  • Claves asociadas a su dato correspondiente.

  • Los datos tienen que estar obligatoriamente identificados unívocamente.

    • En disco: dirección a página de disco.

    • En memoria principal: un puntero al dato.

Tablas Hash

Estructura de datos que asocia llaves o claves con valores.

  • Búsqueda: operación principal que soporta de manera eficiente; acceso a partir de clave generada.

  • Transforma la clave con una función hash en un número hash donde la tabla hash localiza el valor deseado.

Índices basados en árboles

Estructura de datos en forma de árbol preservando el orden de las claves.

  • Árboles binarios: las claves se van almacenando en forma de árbol balaceado basado en disco o en memoria.

  • A tener en cuenta el tamaño de los índices y el coste de construir.

Índices multidimensionales

Los datos se representan como puntos en un espacio multidimensional.

  • Una dimensión es como una columna en una tabla relacional.

  • Tree-index: índice primario en una clave.

  • Acceso aleatorio: índices secundarios en otras claves.

  • Ventajas en las búsquedas: puntos y consultas sobre rangos.

Problema Soluciones
Maldición de la dimensionalidad Partición y reducción del espacio dimensional
Crecimiento exponencial Comprensión de los datos

Aplicaciones

  • Multimedia.

  • Recuperación de documentos.

  • Biología.

  • Almacenamiento y acceso de datos especiales.

Inverted files

Estructura de datos que se utiliza comúnmente en motores de búsqueda y sistemas de indexación.

  • En lugar de indexar los datos directamente por su valor, se indexan por sus términos o palabras clave.

  • Tablas especiales con filas de la forma: pares “valor, lista de claves”.

  • Dado un valor, devuelve todas las claves correspondientes.

    • Como si fueran índices secundarios, estas claves puede usarse para acceder a los datos correspondientes usando un índice primario.

Funcionamiento

  • Cuando se realiza una consulta que incluye un término, el sistema busca en el archivo invertido ese término y recupera la lista de documentos asociados.

  • Luego, puede utilizar esta lista para recuperar los documentos relevantes de la base de datos

Join índices

Los índices de unión son estructuras de datos utilizadas para optimizar consultas que involucran operaciones de unión entre tablas.

  • Los índices almacenan información pre-calculada sobre las relaciones entre las tablas para acelerar el proceso de unión.

  • Permite recuperar rápidamente los registros relacionados sin tener que realizar un escaneo completo de las tablas.

Tecnologías de almacenamiento escalable

Escalabilidad

Capacidad de adaptación y respuesta de un sistema con respecto al rendimiento de este a medida que aumentan de forma significativa el número de usuarios.

  • Hace referencia a la propiedad de aumentar la capacidad de trabajo de un programa sin alterar la eficiencia de trabajo o tamaño de un sistema y sin poner en riesgo el buen funcionamiento y la calidad del mismo.

  • Es un aspecto complejo e importante del diseño.

  • Influye en el rendimiento de forma significativa.

  • Si una aplicación está bien diseñada, la escalabilidad no construye un problema y se deduce de la implementación y del diseño general del sistema.

  • Supone un factor crítico en el crecimiento de un sistema.

Tipos según dimensiones

  • En carga: un sistema distribuido nos hace fácil el ampliar y reducir sus recursos para acomodar cargas más pesadas o más ligeras según se requiera.

  • Geográfica: aquel que mantiene su utilidad y usabilidad, sin importar que tan lejos estén sus usuarios o recursos.

  • Administrativa: no importa qué tantas organizaciones diferentes necesiten compartir un solo sistema distribuido, debe ser fácil de usar y manejar.

Escalabilidad vertical

Escalar hacia arriba un sistema significa una migración de todo el sistema a un nuevo hardware que es más potente y eficaz que el actual.

  • Una vez se ha configurado el sistema futuro, se realizan una serie de validaciones y copias de seguridad y se pone en funcionamiento.

  • Las aplicaciones que estén funcionando bajo la arquitectura hardware antigua no sufren con la migración, el impacto en el código es mínimo.

Ventaja Desventaja
Nivel estructural: no supone ninguna modificación reseñable, lo que la convierte en una buena opción si los costos anteriores son asumibles Limitación de hardware: llegará un momento que existirá algún tipo de limitación hadware
Costos: a medida que se invierte en hadware de muy altas prestaciones, los costos se disparan

Escalabilidad horizontal

Consiste en potenciar el rendimiento del sistema desde un aspecto de mejora global, a diferencia de aumentar la potencia de una única parte del sistema.

  • Se basa en la modularidad de su funcionalidad.

  • Suele estar conformado por una agrupación de equipos que dan soporte a la funcionalidad completa.

Ventaja Desventaja
No existen limitaciones: se pueden agregar recursos de manera infinita Gran modificación: conlleva un gran trabajo de diseño y reampliación
Si la lógica se ha concebido para un único servidor, es probable que se tenga que reestructurar el modelo arquitectónico para soportar este modelo

Balance de carga

Técnica para dividir el trabajo a compartir entre varios procesos, ordenadores, u otros recursos.

  • Está muy relacionada con los sistemas multi-proceso, que trabajan o pueden trabajar con más de una unidad para llevar a cabo su funcionalidad.

  • Para evitar cuellos de botella, el balance de la carga de trabajo se repite de forma equitativa a través de un algoritmo que estudia las peticiones del sistema y las redirecciona a la mejor opción.

  • Hardware: rápidos porque es acceso hardware.

  • Software: más lentos porque simulan el balanceo hardware.

Operaciones

Replicación

Se basa en tener varias copias de un mismo dato en varios nodos de una base de datos distribuida.

  • Síncrona.

  • Asíncrona.

Desafíos
Anomalía por lectura después de escritura

Solución: consistencia read-after-write

  • Leer información editada propia en vez de en las replicas.

  • Seguimiento de la última actualización con timestamps.

Anomalía por lectura de réplica desactualizada

Solución: lecturas en secuencia en la que los datos no tienen una fecha anterior a una lectura previa.

Anomalía por impedancia en la velocidad de réplicas

Solución: si una secuencia de escrituras ocurre en un cierto orden, cualquiera que lea esas escrituras las verá aparecer en el mismo orden.

Particionamiento

Se basa en la idea de tener particionada la base de datos distribuida en varios nodos.

  • Un dato está presente solo en la parte que le corresponde.

  • En el escalamiento horizontal, cuando la computación distribuida se usa para separar la carga entre múltiples servidores.

  • Enfoque tradiciona: la partición horizontal se una base de datos consiste en dividir una o más tablas por filas, generalmente dentro de una única instancia de un esquema y un servidor de base de datos.

  • Sharding: divide las tablas problematicas de la misma manera, pero lo hace a través de potendiales instancias múltiples del esquema.

Ventajas

  • La carga de búsqueda para la gran tabla particionada ahora se puede dividir entre varios servidores.

  • División tanto a nivel de servidor lógico como físico.

Opciones híbridas

La partición se combina con la replicación para que las copias de cada partición se almacenen en varios nodos.

  • Aunque cada registro pertenece exactamente a una partición, aún puede almacenarse en varios nodos diferentes para mejorar la tolerancia a fallos.
