Business Intelligence

Los silos y el caos de la información

  • Los sistemas transaccionales no tienen como objetivo responder a necesidades analíticas, lo cual ocasiona un crecimiento exponencial de informes artesanales, generados desde diversos departamentos pudiendo obtener distintas versiones sobre el dato, métrica o KPIs.

  • Las limitadas capacidades de los silos de información dificultan encontrar el dato apropiado: los esfuerzos para ello, las dependencias de terceros y otros factores son un freno a las oportunidades del negocio.

Credibilidad

  • Los datos no se extraen bajo las mismas pautas de tiempo.

  • Diferentes algoritmos de análisis.

  • Diferentes niveles de extracción.

  • Datos externos no homogéneos.

  • No hay una fuente común de datos.

Productividad

  • Localizar y analizar los datos para los informes.

  • Recoger los datos necesarios a través de diversas tecnologías.

  • Obtener recursos humanos para realizar el informe.

Información

  • Falta de integración y homogeneización: la toma de decisiones requiere información dispersa por todos los sistemas.

  • No hay suficientes datos históricos: el horizonte de datos históricos es diferente para cada aplicación operacional.

Triángulo de información de Abate

  • El Business Intelligence tradicional proporciona informes de “espejo retrovisor”; análisis de datos estructurados para describir las tendencias del pasado.

Business Intelligence

  • Comprende las estrategias y tecnologías utilizadas por las empresas para el análisis de datos empresariales.

  • Su objetivo es permitir el análisis y la interpretación de los datos históricos empresariales, de forma directa y reduciendo las dependencias.

  • Las estrategias implícitas proporcionan vistas históricas, actuales y predictivas del negocio.

  • La identificación de nuevas oportunidades y la implementación de una estrategia efectiva basada en información puede brindar a las empresas una ventaja competitiva en el mercado a largo plazo.

Data Warehouses

Sistema para recopilar y administrar datos de diversas fuentes para proporcionar información empresarial significativa.

  • Un almacén de datos se usa normalmente para conectar y analizar datos comerciales de fuentes heterogéneas.

  • El almacén de datos es el núcleo del sistema de Business Intelligence que está diseñado para el análisis y la generación de informes de datos.

  • Históricas: recopilación de los principales hechos del pasado.

  • Actuales: los últimos datos según el evento deseado; cargas periódicas.

    • Procesos por lotes.

    • Próximos al tiempo real.

    • En tiempo real.

Características

  1. Orientación al tema: ofrece información destinada a un tema específico en vez de la operativa diaria de toda la organización.

  2. Integrado: un almacén de datos se desarrolla combinando datos de múltiples fuentes heterogéneas.

  3. Historización: brindan información en un cierto punto histórico de tiempo.

  4. No volátil: los datos anteriores no se omiten cuando se le agregan datos más nuevos.

Modelado

Hechos

Son los datos que se desean analizar o medir.

  • Representan eventos o transacciones específicas que ocurrieron en un momentos particular.

  • Los hechos son las métricas cuantificables que se almacenan en un almacén de datos y sobre las cuales se realizan consultas y análisis.

Dimensiones

Son las características o atributos que describen los hechos y proporcionan contexto a los mismos.

  • Son los elementos que se utilizan para organizar, filtrar y agrupar los datos en función de diferentes criterios.

Nivel técnico

  • Data Warehouse extrae periódicamente datos de las aplicaciones y sistemas empresariales.

  • Los datos pasan por procesos de formateo e importación para que coincidan con los datos que ya están en el almacén.

  • El almacén de datos almacena estos datos procesados para que estén listos para que accedan las personas que toman decisiones.

La frecuencia con la que ocurren las extracciones de datos, o cómo se formatean los datos, variará según las necesidades de la organización.

Almacenamientos

On Line Transacción Processing

Captura y mantiene los datos de las transacciones en una base de datos relacional.

  • Cada transacción implica registros de base de datos individuales formados por múltiples campos o columnas.

  • El énfasis está en el procesamiento rápido, porque las bases de datos OLTP se leen, escriben y actualizan con frecuencia.

  • Si una transacción falla, la lógica del sistema integrada darantiza la integridad de los datos.

On Line Analytical Processing

Procesamiento analítico en línea.

  • Es el método más utilizado para analizar y evaluar los datos.

Diferencia entre OLAP y Data Minig

  • OLAP y DataMining se utilizan para resolver diferentes tipos de problemas analíticos.

  • Con OLAP se resumen los datos y se hacen pronósticos.

  • Data Mining descubre patrones ocultos en los datos y opera a un nivel detallado en lugar de resumen.

OLAP != Data Mining.

COMPARATIVA OLTP y OLAP

OLTP proporciona un registro inmediato de la actividad comercial actual mientras que OLAP genera y valida información a partir de esos datos a medida que se compilan a lo largo del tiempo.

La perspectiva histórica permite un pronóstico preciso, pero como con toda la inteligencia empresarial, los conocimientos generados con OLAP son tan buenos como lo es la canalización de los datos de origen.

Relational On Line Analytical Processing

  • Aproximaciones conceptuales al modelado de cubos sobre bases de datos relacionales.
Proceso

  1. Se puede interpretar una tabla/fichero.

  2. Se puede identificar sus atributos y agruparlos según su contenido.

  3. Se puede establecer relaciones entre los atributos identificados.

  4. Incluso se puede enriquecer o fabricar nuevos atributos.

  5. Se amplía el modelo repitiendo los pasos anteriores.

Star schema

  • El esquema consta de una tabla de hechos central rodeada de varias tablas de dimensiones.

  • La tabla de hechos contiene las métricas numéricas o medidas clave que se están analizando.

  • Las tablas de dimensiones contienen atributos descriptivos que proporcionan contexto a las métricas en la tabla de hechos.

  • Las relaciones entre la tabla de hechos y la tabla de dimensiones son simples y directas.

Ventajas Desventajas
Sencillos de generar sin conocimiento profundo No pueden incorporar bajo un único modelo toda la problemática empresarial, siendo necesarios tantas estrellas como necesidades de análisis (silos)
Sin restricciones de volumetría o dimensiones Dependiendo del cambio o actualización puede requerir regeneración completa de la dimensión
Flexibilidad en tipos de cargas
Snowflake

  • Es una extensión de la arquitectura Star que se utiliza cuando se necesitan niveles adicionales de normalización para mejorar la eficiencia de almacenamiento y rendimiento de consultas.

  • Las tablas de dimensiones se normalizan aún más en subdimensiones, lo que significa que los atributos de las dimensiones se dividen en tablas separadas.

  • Esto crea una estructura en forma de copo de nieve, donde las relaciones entre las tablas de dimensiones pueden tener varias capas de join, lo que resulta en un esquema de base de datos más complejo.

Mejoras Desventajas
Modelo completo e integro, con alta flexibilidad para el mantenimiento, actualización y mejoras (tablas) Diseño y construcción más compleja
Permite tablas agregadas reales y auto-medidas Los tiempos de respuesta inicialmente pueden no ser óptimos
El modelo en sí mismo ofrece información y relaciones entre los datos No todas las herramientas de Business Intelligence que lo implementan sacan su potencial
Starflake

  • Es una combinación de Star y Snowflake, que busca equilibrar la simplicidad de la arquitectura Star con la eficiencia de almacenamiento y rendimiento de consultas de Snowflake.

  • Las tablas de dimensiones se normalizan solo hasta cierto punto para evitar un exceso de normalización que pueda complicar en exceso el esquema de la base de datos.

  • Por lo tanto, las tablas de dimensiones pueden tener algunas subdimensiones normalizadas, pero se evita la complejidad excesiva de múltiples capas de join.

Mejoras Desventajas
Ya no existen silos de información, todo en un mismo lugar, visión de Data Warehouse real Cualquier cambio de un concepto en una dimensión fuerza a su regeneración integral
Las dimensiones son compartidas No facilita la comprensión de la relaciones entre padre e hijos

Hybrid On Line Analytical Processing

  • Soluciones hibridas buscan obtener lo mejor de MOLAP Y ROLAP.

  • La existencia de cubos MOLAP asociados debe venir por necesidades particulares y concretas, criterios de optimización, criterios de seguridad e incluso ante algún reto del negocio.

Estructura

  1. Nivel superior: es la capa del cliente y contiene las herramientas utilizadas para el análisis de datos de alto nivel, la consulta de informes y la minería de datos.

  2. Nivel medio: transforma los datos en una estructura más adecuada para el análisis y consultas complejas.

  • ROLAP: Sistema de administración de base de datos relacionales extendido que mapea las operaciones en datos multidimensionales a operaciones relacionales estándar.

  • MOLAP: implementa directamente los datos y operaciones multidimensionales.

  1. Nivel inferior: contiene el servidor de base de datos que se utiliza para extraer datos de muchas fuentes diferentes.

Tipos

Almacenes de datos empresariales (EDW)

sirve como base de datos central o clave para facilitar la toma de decisiones en toda la empresa.

Beneficios
Acceso a información entre empresas
Capacidad de ejecutar consultas complejas
Habilidad de conocimientos enriquecidos y con visión de futuro

Almacenes de datos operativos (ODS)

  • Se actualiza en tiempo real.

  • A menudo se usa para actividades empresariales de rutina.

  • Se utiliza como fuente para proporcionar datos al almacén de datos empresarial.

  • Los datos de las bases de datos se almacenan en el ODS diariamente o semanalmente.

  • El ODS actúa como un área de preparación para la integración de datos.

  • ODS envía y almacena los datos al EDW, que se utiliza con fines de Business Intelligence.

Data Mart

  • Es un subconjunto de un almacén de datos destinado a brindar soporte a un departamento, región o unidad comercial en particular.

Data Lakes

Nuevo tipo de repositorio de datos para almacenar cantidades masivas de datos sin procesar en su forma nativa, en una sola ubicación.

El concepto se puede comparar con un cuerpo de agua, un lago, donde el agua fluye, llena un depósito y sale.

  • Facilita el almacenamiento y la ejecución de análisis en datos de IoT generados por máquinas.

  • Es un repositorio de datos almacenados en su formato raw generalmente como blob o datos estructurados y no estructurados.

  • Almacenan datos sin procesar con respecto a identificadores únicos asignados a cada elemento de datos y se etiquetan con un conjunto de etiquetas de metadatos.

  • Petabytes de información que llega constantemente de múltiples fuentes de datos.

Utilidad

  • Un entorno para que los científicos de datos realicen minería y análisis de datos.

  • Un área de almacenamiento central de datos en bruto, con mínima información.

  • Almacenamiento alternativo para los datos históricos.

  • Un archivo en línea para registros.

  • Un entorno para ingerir en streaming con identificación automática de patrones.

¿Quién los usa?

  • Los científicos de datos pasan un 80% de su tiempo preparando datos para el análisis en lugar de analizarlos.

  • Juega un papel importante en todas las industrias.

Lagos de datos en la nube

  • Minimizar los gastos de capital para hardware y software.

  • Obtener nuevas soluciones analíticas en el mercado rápidamente.

  • Eliminar los silos de datos consolidando varios tipos de datos en una plataforma única, unificada e infinitamente escalable.

Pantanos de datos

  • Un lago de datos contiene datos de varias fuentes.

  • Sin embargo, sin una gestión y gobierno de datos adecuados, un lago de datos puede convertirse rápidamente en un pantano de datos.

  • Un pantano de datos no es seguro de usar porque nadie está seguro de dónde provienen los datos, qué tan confiables son los datos y cómo deben protegerse.

Causas

Sedimentos naturales

Sedimentos órganicos se descomponen fácilmente y otros de descomponer.

Sedimentos electrónicos

Pueden ser blobs tremendamente grandes con poco o ningún valor analítico.

  • No tener una estrategia de almacenamiento mantenible que cubra tanto los tipos como las ubicaciones de los datos hará que el lago de datos se llene de montones de sedimentos electrónicos.

Contaminación

Es el aporte de sustancias que tienen un efecto adverso en un ecosistema.

  • Debido a que los lagos de datos están diseñados para escalar ampliamente, es una tentación llenarlos con datos de los que no queremos deshacernos.

Podedumbre

  • Con el tiempo, las cosas que una vez se consideraron útiles pueden comenzar a pudrir la materia orgánica.

    • Datos que ya no se están analizando.

    • Idea de análisis parcialmente implementada que se abandonó.

    • Caranterística mal documentada cuyo propósito original se olvidó.

  • Lo elementos de datos en los mensajes de eventos del sistema se pueden renonbrar, agregar o eliminar.

  • Razones de cumplimiento o gestión de riesgos que controlen los datos que debe almacenar.

¿Cómo prevenir el empantamiento de datos?

  • Calidad de datos.

  • Limpieza de datos.

  • Política de gobierno con respecto a las entradas al lago de datos.

  • Política de documentación.

  • Tecnologías con capacidad de explorar esquemas de lo que se almacena, cómo y dónde se aplican las reglas.

  • Auditar regularmente los metadatos.

  • Si los metadatos difieren de lo esperado, un administrador de datos debe investigar.

  • Entradas de control.

  • Estrategia de dragado y purga: archivar datos que no está utilizando.

¿Cómo construir un catálogo de datos?

1. Recogida de datos

  • La recopilación de metadatos de todos los datos presentes es el primer paso en la construcción de un catálogo de datos.

  • Un catálogo de datos rastrea todos los datos presentes en el repositorio y copia los metadatos en el catálogo.

  • Estos metadatos se utilizan para identificar conjuntos de datos, tablas de datos y archivos.

2. Construcción de un diccionario de datos

  • Una vez recogidos los metadatos, el siguiente paso es construir un diccionario de datos que contenga la descripción de todos los metadatos.

  • Lo ideal es alinearlo a los términos del Business Glossary.

3. Perfilar los datos

  • El siguiente paso en la creación de un catálogo de datos es perfilar los datos disponibles, lo que ayuda a las partes interesadas a visualizar y comprender los datos.

  • Necesario involucrar tanto los Data Owners como a los Data Stewards.

4. Marcar la relación entre los datos

  • Marcar la relación entre los conjuntos de datos es fundamental.

  • Ayuda a las partes interesadas a entender fácilmente la relación de los datos y las fuentes de datos.

  • Esta relación puede ser marcada manualmente por individuos o por sistemas avanzados, que aprenden de las consultas realizadas entre conjuntos de datos o valores similares.

Cuidado con los automatismos, siempre es mejor controlar el proceso si los datos son sensibles y valiosos.

5. Construcción del linaje

  • Los diferentes procesos que intervienen en el flujo de datos se explican a través del linaje.

  • Esto permite a las partes interesadas encontrar la causa raíz de un error muy fácilmente con sólo seguir el linaje.

  • El linaje puede ser horizontal o vertical.

  • Análisis de Impacto: La lectura inversa del linaje da pistas de auditoría.

6. Organización de los datos

  • Los datos dentro de un archivo o una tabla se presentan de forma técnica.

  • Es necesario realizar un esfuerzo manual para organizar los datos de manera que puedan ser fácilmente comprensibles y fiables para los usuarios de la empresa.

  • Algunas de las formas de hacerlo son el etiquetado de los datos, la organización de los datos en función del uso y la función del usuario, y su automatización.

7. Asegurar y trazar el acceso

  • Para un mayor uso del catálogo de datos, este debe ser fácilmente accesible dentro de la pila de datos.

  • Puede estar en un portal interno o en alguna herramienta siempre que toda la seguridad de acceso sea activa.

No todo el mundo puede acceder a ellos y quien lo hace, tiene que ser trazado.

Comparativa Data Warehouses vs. Data Lakes

Detalles técnicos

Características Data Warehouse Data Lake
Datos Relacional desde sistemas transaccionales, bases de datos operativas y aplicaciones de línea de negocio No relacional y relacional desde dispositivos IoT, sitios web, aplicaciones móviles, redes sociales y aplicaciones corporativas
Esquema Diseñado antes de la implementación de Data Warehouse (esquema en escritura) Escrito en el momento del análisis (esquema-enlectura)
Precio/Rendimiento Resultados de consulta más rápidos con almacenamiento de mayor costo Los resultados de las consultas se aceleran con almacenamiento de información de bajo costo
Calidad de los datos Datos altamente curados que sirven como la versión central de la verdad. Cualquier dato que pueda o no ser curado (es decir, raw data)
Usuarios Analistas de negocio Científicos de datos, desarrolladores de datos y analistas de negocios (utilizando datos seleccionados)
Analítica Informes por lotes, Business Intelligence y visualizaciones Aprendizaje automático, análisis predictivo, descubrimiento de datos y creación de perfiles

Ventajas Data Warehouse

  • Permite a los usuarios comerciales acceder rápidamente a datos críticos de algunas fuentes, todo en un solo lugar.

  • Proporciona información coherente sobre diversas actividades multifuncionales.

    • Compatible con informes y consultas ad-hoc.

  • Ingtegra muchas fuentes de datos para reducir el estrés en el sistema de producción.

  • Ayuda a reducir el tiempo de respuesta total para análisis e informes.

  • Permite a los usuarios acceder a datos críticos de varias fuentes en un solo lugar; ahorrando tiempo al usuario al recuperar datos de múltiples fuentes.

  • Almacenan una gran cantidad de datos históricos.

    • Los usuarios pueden analizar diferentes períodos de tiempo y tendencias para hacer predicciones futuras.

Predictivas Data Lakes

  • Los Data Lakes están permitiendo la democratización de la ciencia de datos y el descubrimiento de datos prometiendo respuestas a nuevas preguntas para un grupo mucho más amplio de usuarios en todas las organizaciones.

Desafíos

  • Gestionar el caos.

  • Islas desiertas de datos.

  • Riesgo de seguridad y control de acceso.

  • Pantano de datos.
